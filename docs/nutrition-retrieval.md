# Nutrition Retrieval Audit

Last updated: 2026-06-12.

## Current Runtime Call Chain

The production path is a controlled LangGraph flow, not an open-ended agent:

1. `app.graph.graph.get_compiled_graph()`
2. route-specific parser node, for example `parse_text_meal`, `recognize_dish_photo`, or `recognize_packaging`
3. `app.graph.nodes.nutrition_retriever.retrieve_nutrition`
4. `NutritionRetriever.lookup`
5. `app.tools.nutrition_tools.NutritionSourceRouter.retrieve_candidates`
6. provider adapters: FatSecret, USDA FoodData Central, Open Food Facts, local fallback
7. deterministic candidate ranking
8. `NutritionCandidate.to_per_100g()`
9. `app.graph.nodes.calculator.calculate_totals`

The LLM may extract foods and portion ranges, but calories and macros are not generated by an LLM. The calculator scales source per-100 g values by `grams_min` and `grams_max` in Python.

## Previous Behavior

Before this audit, `retrieve_nutrition` used:

- local fallback table first for most ingredients;
- USDA FoodData Central only after local fallback missed;
- Open Food Facts only for packaged input first, or generic text after USDA missed;
- a generic mixed-food fallback when everything failed.

USDA existed, `USDA_API_KEY` was loaded through `Settings.usda_api_key`, and the key was not hardcoded. However, USDA was often bypassed at runtime by local fallback hits and only used `GET /foods/search`. The detail endpoint `GET /food/{fdcId}` was not used. USDA ranking selected the first usable search row rather than scoring candidates. Provider failures were swallowed gracefully, but some older exception logging could include URL text from HTTP exceptions, so the new code logs sanitized provider/status information instead.

Open Food Facts existed as a live packaged-product source with a file cache. Its public API has strict rate limits, including documented limits for product reads and search, so it remains a packaged-product fallback rather than a broad search source.

## Implemented Architecture

The retrieval subsystem now normalizes all provider outputs into `NutritionCandidate`:

- source and source ID;
- name, brand, food type, region, language, description;
- serving description and metric serving fields;
- source serving nutrient values;
- derived `values_per_100g` only when safe;
- match score and score components;
- compact metadata.

`NutritionPer100g` remains the calculator input. A candidate must have complete per-100 g calories, protein, carbohydrate, and fat before it can feed the calculator.

The router distinguishes:

- generic ingredient;
- branded or packaged product;
- restaurant menu item;
- standard prepared dish;
- user-described composite meal;
- photo-derived food description.

Provider priority:

- generic ingredient: USDA Foundation/SR Legacy/FNDDS, FatSecret generic foods, local fallback;
- standard prepared dish: USDA FNDDS first, then FatSecret, then local fallback;
- branded or packaged product: FatSecret, USDA Branded, Open Food Facts, local fallback;
- restaurant menu item: FatSecret first, then USDA, then Open Food Facts/local fallback;
- composite or photo-derived meals: retrieve candidates for extracted items and keep deterministic aggregation.

## USDA FoodData Central

Source: https://fdc.nal.usda.gov/

API guide: https://fdc.nal.usda.gov/api-guide

USDA FoodData Central documents `GET /food/{fdcId}` and `GET`/`POST /foods/search` under `https://api.nal.usda.gov/fdc/v1`. The API guide also states that API keys must not be made public and documents rate-limit behavior. FoodData Central data are public domain / CC0, with requested citation to USDA FoodData Central.

The current adapter:

- reads only `USDA_API_KEY`;
- never hardcodes or logs the key;
- uses `GET /foods/search`;
- uses `GET /food/{fdcId}` for top search results in the runtime path;
- chooses data types by query kind:
  - `Foundation`, `SR Legacy`, `Survey (FNDDS)` for generic ingredients;
  - `Survey (FNDDS)` first for prepared/composite dishes;
  - `Branded` first for branded/restaurant queries;
- parses calories, protein, fat, carbohydrate, fiber, sugars, and sodium;
- uses source-provided per-100 g values;
- uses source serving weights only as serving metadata unless the calculator already has grams from parsing.

## FatSecret Platform API

OAuth docs: https://platform.fatsecret.com/docs/guides/authentication/oauth2

Food search docs: https://platform.fatsecret.com/docs/v1/foods.search

Food details docs: https://platform.fatsecret.com/docs/v5/food.get

The FatSecret adapter uses only:

- `FATSECRET_CLIENT_ID`;
- `FATSECRET_CLIENT_SECRET`.

Authentication uses OAuth 2.0 Client Credentials:

- token endpoint `https://oauth.fatsecret.com/connect/token`;
- HTTP Basic authentication;
- `grant_type=client_credentials`;
- `scope=basic`.

The access token is cached in memory with an expiry safety margin and a lock around refresh. The client uses the Basic method endpoint `https://platform.fatsecret.com/rest/server.api` with `foods.search` and `food.get`, requesting JSON responses. It does not assume Premier, barcode, localization, NLP, image recognition, or versioned URL access. Provider errors, authorization failures, rate limits, server errors, timeouts, and malformed JSON degrade to an empty provider result so other providers can continue.

FatSecret raw API responses and tokens are not persisted. Eval serialization redacts FatSecret nutrient values from generated reports.

## Multilingual Normalization

`app.tools.food_query.normalize_food_description` preserves:

- original text;
- detected language;
- normalized text;
- canonical English search query when available;
- brand, restaurant, preparation, quantity, unit, and region.

Examples:

- `жареная куриная грудка 200 г` -> `fried chicken breast`, preparation `fried`, quantity `200 г`;
- `биг мак во Франции` -> `McDonald's Big Mac`, restaurant `McDonald's`, region `FR`;
- `Danone Skyr 850 г` -> brand `Danone`, query kind `branded_product`;
- `борщ со сметаной` -> `borscht with sour cream`.

Brand names are preserved instead of translated.

## Ranking

`app.tools.nutrition_ranking.rank_candidates` scores candidates by:

- exact/near-exact normalized name match;
- source priority for the query kind;
- brand match;
- restaurant match;
- region match;
- metric gram serving availability;
- macro completeness;
- data type or food type suitability.

Score components are stored on each `NutritionCandidate` for debugging.

## Failure Behavior

- Missing FatSecret credentials disable only FatSecret.
- Missing USDA key disables only USDA.
- FatSecret unavailable: USDA, Open Food Facts where applicable, and fallback continue.
- USDA unavailable: FatSecret and fallback continue.
- Rate limits and 5xx errors are logged in sanitized form and do not reach the user.
- No exact match returns lower-confidence ranked candidates when possible.
- No usable source candidate uses `generic_fallback` and adds a warning.
- Millilitres are not silently treated as grams; a candidate without safe per-100 g gram values is not used for deterministic calculation.

## Observability

Provider operations create OpenTelemetry spans when tracing is configured:

- provider;
- operation;
- sanitized query or source ID;
- provider-level result count through structured logs;
- selected candidate source, source ID, and score through structured logs.

Never trace or log:

- FatSecret client secret;
- USDA API key;
- access token;
- Authorization header;
- raw provider responses.

## Open Food Facts

API docs: https://openfoodfacts.github.io/openfoodfacts-server/api/

Open Food Facts remains useful for European packaged foods and barcode/product-label scenarios. Its public API documents strict read/search rate limits and warns against search-as-you-type patterns. The current project keeps the existing live lookup and local file cache for packaged-product fallback. A broader Open Food Facts search rollout should add stronger rate limiting and cache policy review before using it heavily.

## ANSES-CIQUAL 2025

ANSES overview: https://www.anses.fr/en/content/ciqual-nutritional-composition-table

2025 dataset: https://entrepot.recherche.data.gouv.fr/dataset.xhtml?persistentId=doi%3A10.57745%2FRDMHWY

CIQUAL is a strong future local source for foods commonly consumed in France and Europe. The 2025 OpenData release includes downloadable XML/XLSX files and describes 3,484 foods and 74 constituents. It was not imported in this iteration because FatSecret and USDA were mandatory and CIQUAL needs a dedicated import/schema refresh workflow. Recommended next step: add a reproducible importer into SQLite with source version metadata, French names, English aliases where available, and local text search.

## Known Limitations

- Query normalization is deterministic and intentionally small; ambiguous multilingual dishes still need better alias coverage.
- FatSecret localization is not assumed. Russian queries are searched with canonical English terms when available.
- Restaurant matching depends on provider coverage and deterministic brand/restaurant signals.
- Composite meals still rely on parser decomposition and portion estimates; retrieval does not solve portion uncertainty.
- Open Food Facts caching predates this audit and should be reviewed before heavier use.
- No large local CIQUAL database is committed.
